## 0.1 第2段階（サーバー実装）でのテストへの影響

本テスト文書群は第1段階（モック）を前提に作成した。第2段階では実サーバー接続に置換され、以下がテスト条件・期待結果に影響する。各 TC 実施時はこの差分を上書き適用する（正本は [server-design.md](../server-design.md)）。

| 観点 | 第1段階（旧・文書の記述） | 第2段階（現行・こちらが正） |
|---|---|---|
| **前提環境** | python 画像サーバー :8000 を起動 | **DB(Apple Container: `./scripts/dev-server.sh`) + `:server` を起動**。既定 :8090。**画像も `:server` の `/images` 配信に統合**（python 画像サーバー :8000 は廃止）。停止時は商品も画像も出ない |
| **データ永続** | 非永続（再起動で初期化） | **PostgreSQL に永続**（ユーザー単位）。状態依存 TC は 1 セッション内完結の制約は不要 |
| **認証** | 任意のメール/パスワードで成功 | **実認証**。要登録。同一メール二重登録=409、誤パスワード=401、空欄=検証エラー(400) |
| **初期データ** | シード注文/住所/支払いが存在 | **新規アカウントは空**（住所/支払い/注文なし）。カタログのみシード。チェックアウト前に住所・支払いの登録が必要 |
| **cart/wishlist/orders/account** | ローカルで即時 | **要ログイン**（サーバー保存）。未ログインはローカルのみで、ログイン時にサーバー内容へ置換 |
| **注文金額** | クライアントが `Totals.kt` で計算 | **サーバーが権威計算**（同一ロジック）。client は住所/支払いの選択 ID のみ送る |
| **注文番号** | `ord_1` などの連番 | `ord_<8桁hex>`（例 `ord_03fd6276`）。日付はサーバー時刻(Asia/Tokyo) |
| **接続ポート** | :8000（画像） | :8090（既定、設定可能。`sutec.server.port` / `PORT`） |

送料規則（3,000円以上無料/未満500円/税込 taxYen=0）・お気に入り同期・i18n・スナップショット保持は第2段階でも不変。

**自動テストによる補完（第2段階）**: サーバー API 層は自動統合テスト `ApiIntegrationTest`（`server/src/test`、実 PostgreSQL に対し `/api/v1` を通し検証。カタログ/認証(201/409/401/400)/カート/カートマージ/注文の権威計算/お気に入り、計12件）＋単体 `OrderTotalsTest` でカバー済み（`./gradlew :server:test`、Apple Container の Postgres で全PASS）。本テスト文書群の手動 TC は、**UI・画面遷移・状態同期・i18n など端末側の確認**に重点を置き、API のレスポンス契約・エラーコード・金額計算は自動テストに委ねてよい。第2段階で新規追加されたクライアント機能（トークン永続化・401自動ログアウト・ゲストカート統合・ロード失敗の再試行・無限スクロール・ダークテーマ）とサーバーハードニング（JWT_SECRET fail-fast・CORS許可制・認証レート制限）の観点は [test-viewpoints.md §8](test-viewpoints.md#8-第2段階で追加されたテスト観点) に追記した。

## 1. 目的

モックアップ第1段階として実装した全 10 機能が、**iOS / Android の両プラットフォーム**で設計どおりに動作することを確認する。特に以下を重点的に検証する。

1. 主要業務フロー(閲覧 → 検索 → 詳細 → カート → チェックアウト → 注文確定 → 履歴反映)が通しで成立すること。
2. 画面横断の状態同期(カートバッジ・お気に入り・言語切替)が即時かつ正確に反映されること。
3. 業務ルール(送料・税込・注文スナップショット)が仕様どおり計算・保持されること。

## 3. テスト方針

### 3.1 テストの種類とレベル
自動テストは未整備のため、本計画は**手動テスト(マニュアル)を主体**とする。

| レベル/種別 | 実施 | 内容 |
|---|---|---|
| 機能テスト | ○ | 各画面・各機能の単体的な動作確認 |
| シナリオ(業務フロー)テスト | ○ | 機能をまたぐ通し確認 |
| 状態同期テスト | ○ | カート/お気に入り/言語の画面横断反映 |
| 探索的テスト | ○ | 観点マップに基づく境界値・異常系の掘り下げ |
| 国際化(i18n)テスト | ○ | 日英切替の網羅・翻訳漏れ確認 |
| クロスプラットフォーム | ○ | Android / iOS の同一シナリオ実施 |
| 回帰テスト | △ | 修正時に影響範囲のシナリオを再実施 |
| 非機能(性能/負荷/セキュリティ) | × | 本段階対象外 |

### 3.2 テスト設計技法
- **同値分割・境界値分析**: 送料無料の境界(小計 3,000 円前後)、価格帯フィルタ境界、数量ステッパーの下限(1→0 で削除)など。
- **状態遷移**: 注文ステータス、認証状態(未ログイン↔ログイン)、カート(空↔商品あり)。
- **画面遷移網羅**: [概要設計書 4.2](../overview-design.md#42-主要画面遷移) の遷移図を基準に主要経路を網羅。

### 3.2.1 イレギュラー(異常系・境界・例外)テストの強化【重点】
正常系だけでなく、以下のイレギュラー領域を重点的に検証する。詳細は [シナリオ §14](test-scenarios.md#14-異常系境界例外強化) / [手順書 §14](test-procedures.md#14-異常系境界例外強化)。

- **入力バリデーション**: 空/空白のみの資格情報でのログイン失敗、必須未入力、超長文・特殊文字・絵文字、非数字・過去日付など不正値。
- **境界値**: 送料 2,999/3,000/3,001 円、価格帯フィルタ境界、数量下限・大量数量。
- **空・未登録状態**: 住所/支払いの全削除・デフォルト削除時のチェックアウト挙動、0件・空一覧表示。
- **状態・並行・遷移(重点)**: 確定ボタン連打による二重注文防止、**選択中の住所/支払いを削除した際の整合性(潜在リスク領域=★)**、処理中の戻る/言語切替、未ログイン注文、ログアウト時のデータ扱い。
- **ネットワーク/ライフサイクル**: 画像取得失敗時の継続動作、通信擬似 delay 中の操作、バックグラウンド復帰・画面回転・再起動での状態。

> ★の「選択中の住所/支払いを削除 → 確定ボタンがサイレント失敗し得る」ケースは、実装上の潜在リスクとして最優先で確認する。

### 3.3 テスト環境

| 項目 | 内容 |
|---|---|
| Android | Android エミュレータ。ビルド `./gradlew :composeApp:assembleDebug` → adb install。 |
| iOS | iOS シミュレータ。`cd iosApp && xcodegen generate && xcodebuild ... -sdk iphonesimulator CODE_SIGNING_ALLOWED=NO` → `simctl install/launch`。 |
| 画像サーバー(第1段階) ※第2段階は §0.1 参照 | 商品画像はローカルの `mock-server`(`python3 -m http.server 8000`)から配信(Android=`10.0.2.2:8000` / iOS=`127.0.0.1:8000`)。**テスト実施前に起動が必須**(停止時は全画像がプレースホルダ)。インターネット接続は不要。 |
| データ | 起動時のシード(商品/カテゴリ/レビュー/初期住所・支払い・注文)を初期状態とする。 |

### 3.4 対象デバイス(代表)
- Android: エミュレータ(標準的な電話サイズ)1 種以上。
- iOS: iPhone シミュレータ 1 種以上。
- ※実機・多解像度網羅は本段階では実施しない。

## 5. 開始基準・終了基準

### 5.1 開始基準(Entry Criteria)
- Android エミュレータ / iOS シミュレータの双方でアプリが起動する。
- テストシナリオ一覧・手順書がレビュー済みである。
- 画像サーバー(`mock-server` の `python3 -m http.server 8000`)が起動している(商品画像の表示に必要)。※第2段階は `:server` 起動(画像も同サーバーの `/images` 配信)が前提。§0.1「前提環境」参照。

### 5.2 終了基準(Exit Criteria)
- **優先度「高」のシナリオを両 OS で 100% 消化**する。合否は下記の★例外を除きすべて合格。
- 優先度「中」以上の未解決の重大不具合(致命/重大)が 0 件。
- 残存する軽微不具合は影響と対応方針が合意されている。

#### ★(潜在リスク検証)ケースの扱い ※本基準の必須の例外
★ TC-151/152(および TC-158)は期待結果を「あるべき挙動」に置くが、現行実装は**サイレント失敗する既知不具合**([レビュー記録 F1](test-review-record.md))を含むため、素直に実施すると **Fail が確定**し「高 100% 合格」と両立しない。これを終了判定に持ち込まないため、以下のいずれかで扱う。

- **分離**: ★ケースは「仕様確認・不具合起票を前提とする検証ケース」とみなし、**合否の消化率・合格率には算入せず**、起票と再現記録をもって「消化済み」とする。または
- **Waive(免責)合意**: 当該既知不具合の影響・修正方針を関係者で合意し、**終了基準上の例外(waive)として明記**したうえで「高 100% 合格」を判定する。

いずれの場合も、★の実挙動(サイレント失敗の有無)は必ず記録し、不具合一覧へ起票する。修正後は通常の合否基準で再判定する。

## 6. 優先度・重大度の定義

### 6.1 テスト優先度

| 優先度 | 定義 | 例 |
|---|---|---|
| 高 | 主要業務フロー・状態同期・金額計算。失敗するとアプリの価値が成立しない。 | 購入通し・カートバッジ同期・送料計算・言語切替 |
| 中 | 補助的機能・CRUD・分岐表示。 | 住所/支払い編集・絞り込み各条件・空状態表示 |
| 低 | 表示微調整・稀な経路。 | レビュー0件表示・タグ配色 |

### 6.2 不具合重大度

| 重大度 | 定義 |
|---|---|
| 致命(Critical) | クラッシュ・主要フロー続行不能・金額誤計算 |
| 重大(Major) | 機能が動作しない・状態同期崩れ・言語切替漏れ(広範) |
| 軽微(Minor) | 表示崩れ・一部翻訳漏れ・軽微な UX 問題 |

## 7. リスクと前提

| 区分 | 内容 | 対応 |
|---|---|---|
| リスク | サーバー(`:server`)依存。未起動/停止で商品・画像とも表示されない(第1段階は画像のみローカルサーバー依存だったが、第2段階は画像も `:server` 配信のため停止時の影響範囲が広がる)。 | 実施前にサーバー起動を確認。画像のみを失敗させたい場合は手順書 TC-160 の `IMAGES_DIR` 起動方法を使う。 |
| リスク・**要注意(第1段階と逆転)** | 第2段階は PostgreSQL に**永続**するため、**アプリ再起動してもデータは初期化されない**(§0.1「データ永続」)。第1段階前提だった「再起動で初期化される」「1 セッション内で状態確認を完結させる」「破壊的 TC 後にアプリ再起動でシードへ復元する」は**すべて成立しない**。 | 状態確認はセッションを跨いでよい。破壊的 TC からの復元は下記ガード(第2段階版)に従う。 |
| 前提・**要注意(第1段階と逆転)** | 第1段階前提「認証はモックで任意情報で成功する」は第2段階では不成立。**実認証**(要登録、重複=409/誤PW=401/空欄=400)に置換(§0.1「認証」)。 | ログイン/ログアウトの状態遷移に加え、登録済みアカウントでのみ成功することと認証エラー系(409/401/400)を確認。 |
| 前提 | iOS は `CADisableMinimumFrameDurationOnPhone=true` が無いと起動時クラッシュ。 | ビルド構成で付与済みであることを起動確認で担保。 |
| リスク | **破壊的 TC がシードを不可逆に破壊する**。TC-102/112(住所/支払い削除)・TC-145/146(全削除)・TC-151/152/158(選択中削除・VM保持中の変化)は**ユーザー単位データ**(住所/支払い/注文)を消す。第2段階は永続のため**アプリ再起動では復元されない**。カタログ(商品)はユーザーに依存しないため破壊されない。 | 下記の実施順序ガード(第2段階版)に従う。 |

#### 破壊的 TC の実施順序・復元ガード(第2段階版)
1. **順序規定**: 住所/支払いの選択・確定を前提とする**非破壊 TC を先に完了**させてから、破壊的 TC(削除・全削除系)を実施する(第1段階と同じ)。
2. **復元手順(第2段階に置換)**: 破壊的 TC は住所/支払い/注文がユーザーアカウントに紐づくため、**専用のテスト用アカウントで実施**し、後続の非破壊 TC(TC-50/103/113 等)は破壊していない別アカウント(またはシードが手つかずの状態)で行う。**アプリの再起動では復元されない**(データは PostgreSQL に永続)。全ユーザーデータを含めた完全なリセットが必要な場合のみ、DB コンテナ自体を再作成する(`container stop sutec-db && container rm sutec-db` の後に `./scripts/dev-server.sh` を再実行。カタログは起動時に再シードされるが、既存ユーザーアカウントは全て消える。所要時間が長く全テスターに影響するため多用しない)。
3. **記録**: 破壊的 TC 実施時に使用したアカウントと、後続 TC が別アカウント/健全なシードから開始したことを実施記録に残す。
